  - kind: prompt-AB
    question: |
      Orbit needs JSON shaped like `{title, summary, tags[]}`. He is
      trying two ways to ask. Which one will give him a parseable
      output more often?
    optionA: |
      ```
      Reply with JSON containing title, summary and tags.
      ```
    optionB: |
      ```
      Reply ONLY with a JSON object matching this schema. Do not
      write text before or after. Do not use code fences. Do not
      comment on the output.
      
      Schema:
      {
        "title": string (max 80 chars),
        "summary": string (max 200 chars),
        "tags": string[] (3 to 6 tags, lowercase)
      }
      ```
    correct: B
    explanation: |
      Option A is ambiguous about things that matter: what are the
      types? Can the model add fields? Can it write "Here is the
      JSON: ..." before? Different models will interpret it
      differently, and the same model will interpret it differently
      across runs.

      Option B does three things: declares an explicit schema (with
      types and limits), forbids the typical wrapping ("text before
      or after", "code fences"), and says what each field is. The
      model has less room to improvise.

      Rule: asking "as JSON" is an invitation to preamble. Asking
      "ONLY a JSON object matching this schema, no text before or
      after" is an executable order.
  - kind: step-order-dnd

  - kind: prompt-anatomy
    task: |
      Label the parts of a well-built structured-output spec. The
      pieces you cannot skip: schema, constraints, example, and the
      anti-preamble rule.
    parts:
      - id: schema
        label: "Explicit schema (fields and types)"
        text: "Reply with a JSON: `{\"title\": string, \"summary\": string, \"tags\": string[]}`."
      - id: constraints
        label: "Per-field constraints (concrete limits)"
        text: "`title` max 80 chars. `summary` max 200. `tags` between 3 and 6, lowercase, no spaces."
      - id: example
        label: "Example of a well-formed output"
        text: "Example: `{\"title\": \"Deploy failed\", \"summary\": \"The deploy job died on step 3.\", \"tags\": [\"deploy\", \"ci\", \"incident\"]}`."
      - id: noPreamble
        label: "Anti-preamble rule"
        text: "Reply ONLY with the JSON object. No text before or after. No code fences. No comments."
  - kind: prompt-AB
    question: |
      Vega is fighting a model that starts every reply with "Sure,
      here you go...". She has two options to cut the preamble.
      Which one is effective?
    optionA: |
      ```
      Do not start with "Sure" or with polite phrases.
      ```
    optionB: |
      ```
      The first character of your response is `{`. Any text before
      `{` invalidates the response. Start directly with the opening
      brace.
      ```
    correct: B
    explanation: |
      Option A is a negative instruction that names the exact phrase
      you want to avoid. Some models comply, others include it
      anyway ("Sure, here you go..." is almost a reflex in RLHF
      models). You are fighting the bias instead of avoiding it.

      Option B defines a positive, verifiable constraint: the first
      character is `{`. This can be checked by a parser, and gives
      the model a concrete rule it follows better than "do not be
      polite". As a bonus, if the output starts with anything else,
      the validator rejects without ambiguity.

      Rule: to cut preamble, reframe as a condition on the first
      character. "Start with `{`" is stronger than "do not greet".

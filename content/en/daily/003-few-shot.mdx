  - kind: prompt-AB
    question: |
      Orbit is classifying incoming messages as `urgent`, `normal`, or
      `spam`. The model keeps confusing `urgent` with `normal`. You have
      to improve the prompt. What do you add?
    optionA: |
      ```
      To classify better, consider that urgent is any message where the
      sender expresses immediate need, high priority, or potential loss
      if not addressed quickly. Normal is anything else that is not spam.
      Spam is promotional or unsolicited.
      ```
    optionB: |
      ```
      Examples:

      Message: "The server is down, we need it back up now."
      Category: urgent

      Message: "Whenever you can, take a look at ticket #123."
      Category: normal

      Message: "You won an iPhone, click here."
      Category: spam
      ```
    correct: B
    explanation: |
      More abstract explanation rarely fixes classification confusion.
      The model already understands the words "urgent" and "normal" in
      English. The problem is not vocabulary, it is where to draw the
      line between the two.

      Three concrete examples define that line much better than a
      definition. "The server is down" anchors `urgent`; "whenever you
      can" anchors `normal`. The model now has reference points.

      Practical few-shot rule: 3 examples beat 3 paragraphs. If the model
      keeps getting it wrong, do not write another paragraph. Add another
      example, ideally for the borderline case that fails.

    question: |
      You need to classify incoming emails as `invoice`, `inquiry`,
      `complaint`, or `other`. Is 1 example per category enough, or
      should you cover 3 to 5 different cases?
    optionA: |
      ```
      Example:
      
      Email: "Attached invoice 0042 for May services."
      Category: invoice
      
      New email: {{input}}
      Category:
      ```
    optionB: |
      ```
      Examples:
      
      Email: "Attached invoice 0042 for May services." -> invoice
      Email: "How much is the annual plan?" -> inquiry
      Email: "The product arrived broken, I want a refund." -> complaint
      Email: "Late payment, we sent the receipt on Monday." -> invoice
      Email: "Hi, is anyone there?" -> other
      
      New email: {{input}}
      Category:
      ```
    correct: B
    explanation: |
      With one example per category the model learns the central case,
      but it has no references for the edges. What about an email that
      mentions an invoice but is really a complaint? An inquiry that
      could be a complaint? Without examples for those, it has to
      guess.
      
      With 3 to 5 examples covering different cases, the model sees
      the variation: "late payment" is also invoice, "is anyone there?"
      is other, not complaint. The boundary between labels becomes
      explicit.
      
      More examples is not always better: if the 5 you add are nearly
      identical, you teach nothing new. The useful rule is to cover
      different cases, not fill space with repeats.

  - kind: prompt-AB
    question: |
      Forge shows you two versions of a few-shot that classifies
      messages as `action required` or `informational`. It misfires on
      ambiguous messages. Which version fixes it better?
    optionA: |
      ```
      Examples:
      
      Message: "I need you to approve the deploy before Friday." -> action required
      Message: "Yesterday's deploy went out without issues." -> informational
      Message: "Closed ticket #42." -> informational
      ```
    optionB: |
      ```
      Examples:
      
      Message: "I need you to approve the deploy before Friday." -> action required
      Message: "Yesterday's deploy went out without issues." -> informational
      Message: "Closed ticket #42." -> informational
      Message: "Look at this when you can, thanks." -> action required (counter-example of "informational": "when you can" is still a request)
      Message: "Just wanted to share that we closed the client." -> informational (counter-example of "action required": "just wanted to" sounds like a request but is not)
      ```
    correct: B
    explanation: |
      Option A shows the central cases: each label has a clean
      example. It works for clear cases. The problem is the edges,
      where language deceives: "when you can" sounds soft but asks
      for action, "just wanted to share" sounds like a request but is
      informational.

      Option B adds counter-examples labeled explicitly. The line
      "counter-example of informational" marks that the case looks
      like one side but belongs to the other. That tells the model
      where the real cut lies, not the apparent one.

      Rule: a good counter-example is not "the opposite". It is a
      case that any reasonable person would put on the wrong side.
      If you needed 2 seconds to classify it, it is a useful
      counter-example.

  - kind: prompt-AB
    question: |
      Vega has a prompt that generates messages for end users. The
      model sometimes includes code examples when nobody asked. She
      wants to tell it not to write code. Which instruction is less
      risky?
    optionA: |
      ```
      Do not include code in the answer. Do not write code. I repeat,
      NO CODE. No code blocks.
      ```
    optionB: |
      ```
      The answer is in prose, addressed to the end user. If you have
      to mention a technical idea, describe it in words (e.g., "the
      system validates the email format"). Example of the desired
      style:
      
      Question: how do I change my password?
      Answer: Go to Settings > Account > Change password.
      It will ask for the current one and the new one twice.
      ```
    correct: B
    explanation: |
      Option A is the classic negative-instruction trap. "Do not X"
      injects the concept X into the model's context. Some models
      handle it, others generate more X than they would without the
      instruction. You end up asking for what you did not want.

      Option B reframes in the positive: it says what TO do
      ("answer in prose", "describe in words"), shows an example of
      the desired style, and code is excluded by construction. The
      model does not need to "not do" anything, it follows the
      pattern.

      Rule: when you want to avoid something, do not name it as a
      prohibition. Show what you DO want. If you must clarify what
      you do not, do it with a labeled counter-example, not with a
      big "no".
---

Mission of the day: counter-examples teach where the label cuts. Big
prohibitions ("do not do X") often make the model do more X.

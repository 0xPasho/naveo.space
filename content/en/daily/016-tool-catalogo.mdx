  - kind: prompt-AB
    question: |
      Hex shows you an MCP catalog. It has 20 tools, but six are nearly
      identical variants: `get_user`, `fetch_user`,
      `read_user_profile`, `lookup_user`, `user_by_id`, `find_user`.
      The model sometimes invokes the wrong one. What do you do first?
    optionA: |
      ```
      Add a paragraph to the system prompt explaining when to use
      each one. "fetch_user is for X, get_user is for Y..." so the
      model has context.
      ```
    optionB: |
      ```
      Keep ONE tool `get_user` that accepts a `mode` parameter
      (basic / full / by-email). Delete the other five from the
      catalog.
      ```
    correct: B
    explanation: |
      Option A treats the symptom. Adding a "when to use each" paragraph
      stuffs the system prompt with more text, inflates the context,
      and forces the model to reason between six similar names every
      turn. Sometimes it will pick right, sometimes not.

      Option B treats the cause. The problem is not that the model
      fails to understand the descriptions: the problem is that six
      tools cover the same space. Folding them into one with an
      argument that picks the mode removes the decision.

      Rule: when two tools get confused in practice, the answer is
      almost never "explain them better". It is to fold them into
      one, or delete the spare. Curation beats more documentation.

  - kind: step-order-dnd
    task: |
      Your catalog has grown to 25 tools and the model picks wrong
      often. Order the steps to curate it, from top to bottom.
    steps:
      - id: s1
        label: "List the tools and mark when each was invoked in the last 30 days."
        detail: "Without usage data, you do not know which ones are dead weight."
      - id: s2
        label: "Delete the ones that were never used. If nobody calls them, they do not contribute."
        detail: "Catalog plus zero usage equals distraction for the model."
      - id: s3
        label: "Identify groups of tools that cover the same space (similar names, near-identical descriptions)."
        detail: "These usually come from different teams that duplicated without knowing."
      - id: s4
        label: "Fold each group into one tool with a mode or type parameter."
        detail: "One tool with a clear `mode` gives the model one decision, not six."
      - id: s5
        label: "Rename the survivors so the name alone tells the model when to pick it."
        detail: "If the name needs a long description to be distinguished, the name is wrong."
      - id: s6
        label: "Measure again: does the model pick better with a trimmed catalog?"
        detail: "If yes, done. If not, look at the descriptions of the survivors."
  - kind: prompt-anatomy
    task: |
      Label the parts of a good tool name inside a large catalog. The
      intuition: the name should be self-explanatory, namespaced, and
      guide the model on its own.
    parts:
      - id: namespace
        label: "Namespace (which domain it belongs to)"
        text: "`billing.` prefixes every tool in the billing area. Visually separates it from the rest of the catalog."
      - id: verb
        label: "Verb (what it does)"
        text: "`create_` signals that the tool produces a new resource. Not `do_`, not `manage_`, not `handle_`."
      - id: resource
        label: "Resource (what it acts on)"
        text: "`invoice` names the concrete object. Not `record`, not `entity`, not `thing`."
      - id: modifier
        label: "Optional modifier (special case if it deserves one)"
        text: "`_draft` suffix, only if you really need to distinguish variants of the same verb plus resource."

Mission of the day: the catalog is an interface. Five well-named tools

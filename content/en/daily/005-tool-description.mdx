  - kind: tool-description
    toolName: send_alert
    context: |
      Schema for the `send_alert` tool:

      {
        "name": "send_alert",
        "parameters": {
          "type": "object",
          "properties": {
            "severity": { "type": "string", "enum": ["low", "med", "high", "critical"] },
            "message": { "type": "string" },
            "channel": { "type": "string", "enum": ["pager", "email", "chat"] }
          },
          "required": ["severity", "message", "channel"]
        }
      }

      `send_alert` wakes up someone on the on-call shift. It is not a log:
      every invocation costs a real person's time. Write a description
      that clearly states when to invoke it, when NOT to, and what
      happens on invocation.

      The validator looks for three ideas, not exact phrases: that you
      mention something is "urgent" or a synonym, that the "crew" or
      on-call team gets the alert, and that the decision is tied to a
      concrete "incident" and not to hypothetical queries.
    starter: |
      Send an alert.
    requiredPhrases:
      - "urgent"
      - "crew"
      - "incident"
  - kind: prompt-AB
    question: |
      Forge shows you two descriptions for the same `send_alert` tool.
      Which one makes the model pick it correctly when it should?
    optionA: |
      ```
      Sends an alert to the relevant team.
      ```
    optionB: |
      ```
      Wakes up the on-call person when there is an urgent incident that
      needs immediate attention from the crew. Do NOT use for
      hypothetical queries, simulations, or historical events. Use
      severity "critical" only when there is risk of data loss or
      production downtime.
      ```
    correct: B
    explanation: |
      Option A is vague on the three dimensions that matter. "Sends an
      alert" does not say whether it wakes someone or just logs.
      "Relevant team" does not define which one or when. The model has no
      way to pick it with confidence, so it under-uses it or over-uses it.

      Option B names the concrete effect ("wakes up the on-call person"),
      the right trigger ("urgent incident"), the NO cases ("hypothetical
      queries, simulations, historical events"), and guides the use of
      severity. The model can now decide well.

      Rule: the description should say what the tool name does not
      communicate on its own. If your description repeats the name, it
      is not adding anything.

Mission of the day: the description is the only way the model has to
pick your tool. If you do not tell it what happens, when YES, and when
NO, the model does not pick it.

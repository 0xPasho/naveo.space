  - kind: prompt-AB
    question: |
      Orbit asks for help with a prompt: the summary has to fit in a
      push notification of at most 50 words. Which of the two
      formulations gives the output they need?
    optionA: |
      ```
      Try to be brief and concise. The summary is for a push
      notification, so keep it short.
      ```
    optionB: |
      ```
      MAX 50 words. Cut rather than go over. If it does not fit in 50,
      drop secondary details. Return only the summary, do not count
      words at the end.
      ```
    correct: B
    explanation: |
      Option A is a soft constraint: "try", "concise", "short". The
      model will read that as a suggestion and may return anywhere from
      30 to 120 words depending on content. Some fit in the
      notification, others do not.

      Option B is hard: it gives the exact number, says what to do if it
      does not fit ("cut rather than go over", "drop secondary
      details"), and prevents a common side effect ("do not count words
      at the end" stops the model from appending "Total: 47 words"
      below).

      Rule: if the constraint is operational (it fits or it does not),
      it has to be hard. If it is a preference (shorter is better, but
      nothing breaks if it is long), it can be soft.

  - kind: prompt-AB
    question: |
      You told the model: "MAX 30 words, do not go over for any
      reason". The original report has a critical safety detail that
      does not fit in 30 words. When does a hard constraint become
      counterproductive?
    optionA: |
      ```
      Keep the hard constraint of 30 words. If it does not all fit,
      let the model decide what to cut.
      ```
    optionB: |
      ```
      Raise the limit to 60 words, or add an exception: "MAX 30 words
      EXCEPT if there are safety details, in which case mention them
      fully."
      ```
    correct: B
    explanation: |
      A hard constraint is useful when the rule is reasonable for every
      input. When there is an input where the rule competes with
      something more important (safety, accuracy, compliance), the
      model has two bad options: respect the constraint and omit
      critical info, or ignore the constraint and break your trust.

      When this happens repeatedly, the hard constraint starts failing
      in weird ways: the model "breaks" the cap when it feels
      important, or truncates the critical detail without warning.
      Neither is good.

      Fix: either raise the cap to one that covers the worst case, or
      write the exception explicitly. "MAX X words EXCEPT in case Y" is
      legal and the model respects it better than a universal cap it
      fights with from time to time.
---

Mission of the day: hard constraints are your seatbelt for what is
critical. Soft constraints are for style. If you confuse them, the
model wins and you lose.

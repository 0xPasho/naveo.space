  - kind: prompt-AB
    question: |
      Atlas shows you two descriptions for a `charge_card` tool. There
      is an automatic retry layer above the client. Which description
      keeps the model from firing it twice by accident?
    optionA: |
      ```
      Charges a card. Creates a new charge every time it is invoked.
      ```
    optionB: |
      ```
      Charges a card. NOT idempotent: each invocation creates a new
      charge. If you need a safe retry, pass `idempotency_key` (a
      unique string). With the same key, it returns the previous
      charge instead of creating a new one.
      ```
    correct: B
    explanation: |
      Option A is the classic trap. "Creates a new charge every time"
      sounds explanatory, but it is telling the model (and the client
      that orchestrates retries) that this tool can never be safely
      retried. One network timeout and the user pays twice.

      Option B names the risk ("NOT idempotent") and offers the
      escape: a key you generate and send. With the same key, the
      tool returns the existing charge. The model learns to use the
      key whenever retry is possible, and retries stop being fatal.

      Rule: irreversible side effects (charging, sending email,
      deleting) need to be either idempotent by design or accept an
      idempotency_key. Otherwise a flaky network ruins your day.

Mission of the day: if the tool has a side effect, tell the model whether
retrying is safe. Silence in the description is the most expensive bug source.

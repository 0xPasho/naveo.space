## The schema is NOT just documentation. it's also validation.

When the model invokes a tool, the runtime can:

1. **Accept the invocation** if the arguments pass the schema, and let the handler do its work.
2. **Reject it and return a validation error** if the arguments violate any constraint, and let the model retry with valid values.

The second flow is **way cheaper** than letting the handler receive garbage and deal with it. That's why a tight schema pays double: the model learns the valid values from feedback, and your handler skips checks.

### The constraints that matter

JSON Schema has more tools than people use. The ones you'll use 95% of the time:

| Case | Constraint | Example |
|---|---|---|
| Discrete values | `enum` | `"enum": ["sms", "comms", "email"]` |
| Number in range | `minimum`, `maximum` | `"minimum": 0, "maximum": 100` |
| Integer (not float) | `type: "integer"` | use instead of `"number"` for counts |
| String with bounded length | `minLength`, `maxLength` | `"maxLength": 280` (tweets) |
| String with format | `format` | `"format": "date-time"`, `"format": "email"`, `"format": "uri"` |
| String with regex | `pattern` | `"pattern": "^[A-Z]{3}-\\d{4}$"` (codes like "BAY-1234") |
| Non-empty array | `minItems` | `"minItems": 1` |

### Why it matters: three consequences

1. **The model doesn't invent values.** Without enum, if you ask for `severity`, it'll try "high", "alta", "urgent", "critical". With enum, it sticks to the 3 you defined.
2. **Errors are useful.** "value not in enum" is very different from a handler that crashes with `Cannot read property of undefined`.
3. **Your description gets shorter.** Things like "the format is YYYY-MM-DD" stop being text the model can ignore and become a `format: date-time` that the system actually checks.

### Your task

On the right you have `schedule_maintenance` empty. Define the 5 parameters with the right constraints. The validator checks each constraint with regex; at the end an llm-judge confirms they don't contradict the use case.

> The tighter the schema, the less defensive code in the handler. **The schema is the first line of defense.**

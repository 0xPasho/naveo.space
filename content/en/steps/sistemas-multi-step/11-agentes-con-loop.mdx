## The agent that knows when to stop.

<Character name="orbit" /> has a rule: *any loop without an exit condition is an incident waiting to happen*. If you're going to let the agent run on its own, you have to be able to answer before hitting **Run**: what makes it stop?

An agent that calls tools in a loop is the central piece of any modern autonomous system. Sounds simple: model picks a tool, you execute, send back the result, repeat until done.

The problem: **when does "done" happen?** Three ways to never finish:

1. **Infinite loop.** The model gets "stuck" calling the same tool over and over with the same args.
2. **Long but non-infinite loop.** The model does reasonable things but doesn't converge. 30 turns later you're still burning tokens.
3. **Cascading error.** A tool fails, the model doesn't know, keeps calling things that depended on that result.

Designing good **stop conditions** is what separates an agent you can let run from one that needs constant human supervision.

### The three stop conditions

1. **`max_steps`**. absolute counter. If it hits 10-15 without a `final`, exit with error. Universal safety net.
2. **Loop detection**. if the model requests the same tool with the same args 2-3 times in a row, it's stuck. Exit.
3. **Tool errors are propagated**. when a tool fails, the error goes back to the model as input (`{ ok: false, error: ... }`). The model decides how to recover. The loop **does NOT break**. it only breaks if the model decides to or if the other stop conditions trigger.

### The task

Implement the loop. On the right you have a starter that **works on the happy path** but has no stop conditions.

### Why we don't execute your code

Same principle as Track 3, step 06: an LLM-judge reads your implementation and reasons whether it would correctly handle the 4 scenarios. That means your code has to be **readable**. Clear names, obvious control flow, early returns for errors.

### How it's evaluated

5 LLM-judge criteria:

1. `max_steps` correctly implemented.
2. Loop detection implemented.
3. Tool errors go back to the agent as input, don't terminate the loop.
4. Happy path intact.
5. Structured returns with an `ok` boolean.

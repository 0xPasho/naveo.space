## Two router flavors, two different contexts

In step 08 you wrote an LLM-based router. But another alternative exists: **rules-based router**, a code function with `if/else` on keywords and patterns. When does each one fit?

### Rules: fast, cheap, fragile

Pros:

- Latency <1ms.
- Zero cost per invocation.
- Deterministic. Same input, same route, always.
- Auditable. Anyone reads the code and gets what it does.

Cons:

- Fragile to input variations. A typo, a synonym, a translation and the rule fails.
- Maintenance explodes. Every new case is another branch, and it never ends.
- Doesn't generalize. If the input corpus changes, rules don't learn.

### LLM: flexible, expensive, non-deterministic

Pros:

- Generalizes by meaning, not by keywords.
- Handles synonyms, multilingual, typos for free.
- You can wire the safe default (escalate) into the route description, not into code.

Cons:

- Latency 200-2000ms.
- Cost per invocation.
- Non-deterministic (without temperature=0 and a well-built eval set).

### The decision tree

```
Is the input structured (form, JSON, dropdown)?
  YES → rules. Done.
  NO → continue.

Do you have 2-3 very distinct categories that a simple regex separates?
  YES → rules with a clear fallback.
  NO → continue.

Is critical latency <50ms or do you have thousands of QPS?
  YES → rules + optional escalation to LLM for default cases.
  NO → LLM.
```

> The error <Character name="orbit" /> sees most: teams that start with rules "to save money" and, six months later, have 4000 lines of `if/else` nobody understands, 60% accuracy, and a "special cases" backlog growing faster than it closes.

### Hybrid: the best of both

In serious systems, the winning pattern is usually:

1. **A very fast rules layer** for the obvious cases (95% of real traffic).
2. **An LLM layer** that takes everything rules flag as "not sure".

Rules give low latency on the happy path; the LLM catches the long tail without blowing up costs.

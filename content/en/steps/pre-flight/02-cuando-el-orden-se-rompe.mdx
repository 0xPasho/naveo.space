## The bug that doesn't warn you

In the last lesson the swaps were obvious. *Engage thrusters before close airlock*, and the atmosphere vents into space. Red alarm, everyone screaming. Easy to catch.

But most real bugs aren't like that. They're **silent**. The protocol runs. The ship launches. Every gauge green. And the damage shows up three shifts later, when it's too late to react.

### Anatomy of a silent swap

Take two steps from the launch in lesson 1:

```
Confirm crew on board
Close airlock
```

In this order, "Confirm crew" counts heads while the airlock is still open. If someone's outside, they walk in now. If nobody's outside, you seal with the correct count.

Flip the two:

```
Close airlock
Confirm crew on board
```

The seal runs first. Then you count. The count says "4" because there are 4 people inside, but the ship was supposed to launch with 5. **Four inside, one on the dock, nobody the wiser.**

The protocol never failed. It just stopped protecting against the problem it was designed for.

### The question you need to learn to ask

When something goes wrong and you can't find "the error", the question isn't *"which line is broken?"*. The question is:

> **"Each step, is it still doing what it promised when we wrote it?"**

That question is the difference between a 2-minute debug and a 2-day debug. You'll use it on every prompt bug you write: the prompt "runs", the model "responds", but the response stopped being the one you needed because some earlier step changed what was reaching it.

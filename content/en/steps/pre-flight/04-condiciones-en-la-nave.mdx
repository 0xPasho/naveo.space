## Two paths, one decision

A sequence runs everything top-to-bottom. But sometimes the crew has to **decide**: do A, or do B, depending on what's happening right now.

That's a **condition**. In code:

<DecisionFlow condition="condition?" ifLabel="IF" elseLabel="ELSE" ifAction="action_A" elseAction="action_B" />

In the ship:

- _If the airlock is closed, ignite. else, abort._
- _If oxygen is below 20%, sound the alarm. else, stay silent._
- _If the captain's hand is on the key, fire. else, wait._

The condition is a question with a yes/no answer. The `IF` branch runs when yes. The `ELSE` branch runs when no.

### Watch it run with a real value

Take the oxygen rule and feed it two different sensor readings:

<DecisionFlow condition="oxygen < 20%?" ifLabel="IF" elseLabel="ELSE" ifAction="sound alarm" elseAction="stay silent" />

- Sensor reads **35%** → the condition is **false**. The `ELSE` branch runs: silence.
- Sensor reads **12%** → the condition is **true**. The `IF` branch runs: sound the alarm.

Same structure, two outcomes. the only thing that changed was the value the ship was reading in that instant. The program doesn't "remember" what was happening before; every time it hits the condition, it **measures again**.

### The same shape works for a prompt

When you ask the model for something later, you'll write the exact same decision in plain English:

> *"If the bio mentions an email, return the email. Otherwise, return null."*

Same binary decision, same two branches. The only thing that changed is that the "sensor" is now the input text and the "actuator" is what the model returns to you.

On the right, a single binary call. Read the protocol. Read the situation. Pick.

## What the model can "see"

The model has no memory between calls. **Zero.** It doesn't remember you from prior turns. It doesn't know the session continues. Every call starts from nothing.

### The trick: we resend the whole history each time

So how do conversations "work"? Trick: when you see a chat with multiple messages, what's happening underneath is this:

```
CALL 1
  input:  [user_message_1]
  output: response_1

CALL 2
  input:  [user_message_1, response_1, user_message_2]
  output: response_2

CALL 3
  input:  [user_message_1, response_1, user_message_2, response_2, user_message_3]
  output: response_3
```

Every time you speak, we re-send **the entire history** to the model, as one giant input. The model doesn't "remember": it's re-reading everything from scratch every time.

### The context window

The input has a hard limit. For current models:

- Claude, modern GPT: ~200,000 tokens (a short book)
- Older models: 8,000 to 32,000 tokens
- Cheaper models: less

That's the **context window**: how much text the model can "see" in a single call. If the history exceeds the limit, you have to trim it. The common move is to **drop the oldest** and keep the recent.

### Why it "forgets"

Result: in a very long conversation, the model appears to "forget" things you said early on. But it didn't forget: **the messages never reached it**. They got cut by the trim that fit through the window.

### Practical implications

1. **If you want something important to stick, repeat it periodically** or put it in the system prompt (which almost never gets trimmed).
2. **In batch processing, don't assume the model "remembers" the previous item.** Send it the context it needs on every call.
3. **Long documents** that don't fit the window need to be split into pieces (*chunking*) and processed one at a time.

> This is central to understanding RAG, multi-turn conversations, and why models sometimes seem to contradict themselves: different call, different slice of the history visible.

## How the model "reads": tokens

In lesson 14 you saw that the model predicts token by token. But what exactly is a token?

### A token is NOT a word

It's finer than that. A token is a **piece** the model treats as an atomic unit. It can be:

- a whole word (`"hello"` → 1 token)
- a piece of a word (`"unbelievable"` → about 3 tokens: `un` + `believ` + `able`)
- a symbol or punctuation mark (`","` → 1 token; `"."` → 1 token)
- a space (yes, spaces count too)

Tokenization is a trick to **compress the vocabulary**: instead of one token per possible word (hundreds of thousands), the model learns about 50,000 common pieces and builds the rest by combining them.

### Useful approximations

For English text:

- 1 token ≈ 0.75 words
- 1 word ≈ 1.3 tokens
- 1,000 tokens ≈ 750 words (about 3 paragraphs)

For code:

- Much denser. A short program can use double or triple the tokens of the same amount of prose.

For non-English languages (Spanish, French, Chinese):

- More tokens to say the same thing. The model is trained mostly on English data, so pieces in other languages tend to be shorter (and therefore, more of them).

### Why you care

Three practical reasons:

**Cost.** APIs charge per token (input and output separately). More tokens, more money.

**Limits.** Every model has a max tokens per call (its "context window", which is the next lesson). If you exceed it, it doesn't fit.

**Speed.** The model generates token by token. A 1,000-token response is slower than a 100-token one.

> Practical takeaway: **write short, precise prompts**, ask for concise outputs, and if you're processing a lot of text, split it into manageable chunks.

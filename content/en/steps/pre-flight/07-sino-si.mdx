## When there are more than two options

So far the decisions have been binary: `IF` this, `ELSE` that. Two branches, one decision.

But most real decisions have **more than two possible outcomes**. Think about oxygen:

- 8% → crisis, run every emergency protocol
- 22% → serious warning, prepare
- 40% → low but manageable, notify
- 70% → normal, silence

To express this you chain several `IF`s. It's called **else-if** (in some languages: `else if`, `elif`).

### How to write it

<DecisionChain branches="oxygen < 10% => critical alarm; oxygen < 25% => warning alarm; oxygen < 50% => informational notification" fallback="silence" ifLabel="IF" elseIfLabel="ELSE IF" elseLabel="ELSE" />

The ship evaluates **top to bottom** and stops at the **first one that holds**. Then it jumps straight to the end.

### The key rule: first match wins

Look at what happens with `oxygen = 8%`:

1. Is 8 < 10? **yes** → run `critical alarm` and end. **The rest isn't even evaluated.**

What about `oxygen = 22%`:

1. Is 22 < 10? no, continue.
2. Is 22 < 25? **yes** → run `warning alarm` and end.

And with `oxygen = 72%`:

1. Is 72 < 10? no.
2. Is 72 < 25? no.
3. Is 72 < 50? no.
4. `ELSE` → run `silence`.

The final `ELSE` (with no condition) is the **catch-all**: it runs only when none of the previous conditions held. It acts as a safety net.

### The classic bug: reversed order

If you flip the order of the branches, everything breaks without throwing an error.

```
IF oxygen < 50%:
  informational notification
ELSE IF oxygen < 25%:
  warning alarm
ELSE IF oxygen < 10%:
  critical alarm
```

With `oxygen = 8%`, the first condition (`8 < 50`) is **already true**. The ship runs *informational notification* and bails. The critical alarm never fires.

> Practical rule: in `ELSE IF` chains ordered by threshold, the **most restrictive case** goes first (the smallest range) and you relax from there down.

You'll see this same chain in complex prompts: *"If the bio has email **and** phone, return both. Otherwise, if it has only email, return the email. Otherwise, if it has only phone, return the phone. Otherwise, return null."* Same pattern.

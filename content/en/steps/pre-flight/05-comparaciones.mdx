## How the IF gets computed

In the last lesson you learned `IF / ELSE`. Now one step deeper: how does that condition get computed?

The condition is a **comparison**. Three basic operators cover almost everything you'll see:

| Symbol | Meaning |
| --- | --- |
| `<` | less than |
| `>` | greater than |
| `==` | equal to |

And their close siblings: `<=` (less than or equal), `>=` (greater than or equal), `!=` (not equal).

Each comparison takes two values and returns **true** or **false**. That's all the `IF` needs in order to decide which branch runs.

### The boundary trap

`<` and `<=` look alike but **they are not the same**.

Look at the rule:

```
fuel < 20%
```

With the sensor at:

- **18%** → 18 is less than 20 → **true** → runs `IF`
- **25%** → 25 is not less than 20 → **false** → runs `ELSE`
- **20%** → 20 is not less than 20 (it's equal) → **false** → runs `ELSE`

The boundary case (20%) is the easiest one to misread. The rule `<` is **strict**: it does NOT include the boundary. If you want the boundary to fire too, you write `<=`.

> This detail is responsible for an absurd amount of production bugs. The alarm just "doesn't fire" exactly at the critical point because someone wrote `<` when they meant `<=`. Read them carefully.

### What this looks like in plain language

You'll use comparisons when you ask the model for something, even if you don't write them with symbols:

- *"If the bio is **more than** 500 characters, summarize it."* → `len(bio) > 500`
- *"If the user is **at least** 18 years old, show them section X."* → `age >= 18`
- *"If the response is **exactly** empty, ask it to retry."* → `response == ""`

When the model reads your prompt, it's assembling these comparisons in its head. The more explicit you are about the operator (*"at least", "more than", "exactly"*), the less room you leave for the model to interpret as it likes.

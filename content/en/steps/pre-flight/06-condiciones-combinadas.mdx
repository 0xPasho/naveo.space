## One decision, several conditions

So far the conditions have been simple: one yes/no question. The reality is that most decisions on the ship **combine** two or more checks.

Three operators handle the combinations:

| Operator | Means | Is true when |
| --- | --- | --- |
| `AND` | both | both sides are true |
| `OR` | either | at least one side is true |
| `NOT` | invert | the original side is false |

### AND: both sides have to be green

```
IF airlock closed AND fuel > 50%:
  fire engines
ELSE:
  wait
```

To fire, **both** conditions have to be true. If the airlock is open you don't fire, no matter how much fuel you have. If you have 30% you don't fire, no matter how closed the airlock is.

### OR: one is enough

```
IF door forced OR motion sensor active:
  alert security
ELSE:
  silence
```

To alert, **either** one is enough. It's a more sensitive system, designed for *"I don't want any case to slip past me"*.

### NOT: flipping a condition

```
IF NOT crew_on_board:
  abort launch
```

Read it as *"if there's NO crew on board, abort"*. `NOT` takes a true condition and turns it false (and vice versa).

### The most common bug: confusing AND with OR

`AND` and `OR` look three letters apart but **they build opposite behaviors**.

- `door closed AND alarm active`: very specific. Everything has to happen.
- `door closed OR alarm active`: very loose. As long as something happens.

If you confuse one for the other, one of two things happens: either the system **never fires** (you expected `OR` and wrote `AND`), or it **fires constantly** (you expected `AND` and wrote `OR`). Both ruin your day.

> When you assemble a prompt you'll write this in plain English: *"if the bio has an email **and** a phone, return them. If **only** the email is there, return the email. If there's **none**, return null."* Same pattern, same operators.

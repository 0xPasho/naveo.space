## Input and output: a function's contract

A function is a black box. You feed things in. Different things come out. What matters to understand:

- **Inputs:** what the function needs to do its job. The parameters you pass when you call it.
- **Output:** what the function returns at the end. The result.

```
FUNCTION calcRemainingFuel(distance, weight):
  ...calculations...
  return available_minutes
```

- Inputs: `distance`, `weight`
- Output: `available_minutes`

### Why thinking in input/output matters

When you see a new function, before reading the body, look at the **signature**: name + inputs + output. That tells you 90% of what it does.

Look at this one:

```
formatTimestamp(epoch, timezone) → string
```

Without reading a single line inside, you already know:

- It needs an `epoch` (seconds) and a `timezone` (text).
- It returns a formatted `string`.
- It doesn't modify global variables on the side, at least according to its signature.

That fast reading is what lets you use functions without having to understand every implementation detail.

### Functions with no output (or no input)

Some don't return anything. They just **do** something:

```
soundAlarm("critical", 30)   # inputs: 2. output: none.
```

Others don't need inputs:

```
currentTime()                # inputs: none. output: timestamp.
```

Both are legit functions. But the most useful ones have **at least one of each**.

### The LLM also has a signature

This is the most important idea for the rest of the curriculum:

```
LLM(prompt) → response
```

The model is a function. **Input:** a prompt (text). **Output:** a response (text). That's it. Everything you learned about functions applies to the model:

- The exact same input can produce a different output (it's not 100% deterministic, you'll see why in lessons 17-18).
- Designing the input (the prompt) well is half the job. The function can't invent what you didn't give it.
- The output is only useful if the input covered everything needed.

> **This is the mental foundation for prompting:** you understand the LLM as a function `text → text`. When you get frustrated because *"it doesn't do what I want"*, the first thing to check is always the input.

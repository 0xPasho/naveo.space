## A taxonomy of "missing data"

When a field "isn't there", it can mean many different things. Your prompt needs to treat them all as `null`:

1. **Not mentioned.** The bio simply doesn't include the field.
2. **Mentioned as absent.** *"No contact registered"*, *"pending assignment"*.
3. **Partially formatted.** *"@bru"* (looks like an email but isn't).
4. **Contradictory.** *"No email. bru@thecrew.io"* (the bio contradicts itself).
5. **Placeholder.** *"TBD"*, *"[redacted]"*, *"n/a"*.

A prompt that just says "extract the email" will fail on some of these. The model can:

- Return the literal placeholder (*"TBD"*).
- Invent a plausible one (pure hallucination).
- Return the handle as if it were an email.
- Return empty string `""` instead of null.

### The minimum rule: enumerate the "absence" triggers

Your prompt has to **explicitly list** what counts as absence:

```
If there's no email OR the bio says "no contact", "pending", "TBD", "n/a",
or any similar placeholder, return null.

If the text looks like an email but doesn't have "local@domain.tld" format,
also return null.
```

Four or five trigger words cover 90% of cases.

### Real null, not the string "null"

```json
{ "email": null }     ✓ correct
{ "email": "null" }   ✗ string that says null
{ "email": "" }       ✗ empty string
{ "email": "TBD" }    ✗ placeholder passed to output
```

Telling the model *"return the JSON null VALUE, not the word 'null' as a string"* usually does it. If in doubt, you can also say *"the output must parse with `JSON.parse` and the `email` key must be strictly null or a string"*.

### Why this matters downstream

When a downstream system receives `"TBD"` instead of `null`:

- The filter "all emails containing @" lets it through as valid.
- The attempt to email "TBD" breaks the system.
- The query "count registered emails" returns an inflated count.

Each of these bugs is **invisible on the first test pass** and shows up in production. That's why the null case is one of the most important to cover.

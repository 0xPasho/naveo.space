## Prompt engineering = iteration

Almost no prompt comes out right on the first try. The real skill isn't writing the perfect prompt in one shot. It's **iterating fast**.

### The loop

```
1. Write prompt v1.
2. Run it against your test cases.
3. Look at what fails and why.
4. Change ONE specific thing.
5. Run again.
6. Repeat.
```

The step most people skip: **step 3**. Look at the actual output, don't assume what the problem is. If you assume, you end up changing what isn't broken.

### The 4 failure categories

Almost every prompt bug falls into one of four. Recognizing which saves you hours:

**1. Format.** The content is correct but breaks the parser (extra preamble, missing key, inconsistent format).
**Fix:** add explicit format rules, use `<output_format>`, show the exact output shape.

**2. Coverage.** Some cases work, others don't. The prompt assumes the input takes a certain form and edge cases break it.
**Fix:** add a specific rule for the failing case.

**3. Inventions (hallucination).** Returns data that's not in the input.
**Fix:** add *"if not present, return null"*, explicitly.

**4. Tone / language / length.** The easiest. Specific constraints solve.

### The isolated-change trick

When you iterate, change **one thing at a time**. Temptation: see 3 problems, change 3 rules in the prompt, run everything. Result: if 1 of the 3 fixes makes something else worse, you don't know which it was.

Discipline: one iteration = one change = one run. Boring but a thousand times faster to land at the correct prompt.

### The 3 questions for every failing output

In order:

1. **Which criterion failed specifically?** (The feedback says.)
2. **Does the output show partial understanding or full miss?** (Partial = tweak; full = rethink.)
3. **Does my prompt mention this situation explicitly?** (If "no", you know what to add.)

> The final prompt that passes all cases is rarely the most elegant one. It's the one that survived the 4 iterations it took. Don't fall in love with v1.

On the right, an intentionally weak starter. Run it as-is, look at what fails, change one thing. Repeat.

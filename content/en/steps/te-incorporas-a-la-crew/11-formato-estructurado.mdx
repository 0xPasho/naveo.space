## Beyond simple JSON: nested structures and schemas

You've already written a prompt that extracts a flat object (`alias`, `puesto`, `contacto`). The reality is that many tasks need richer structures:

- **Lists** of objects (several items of the same type).
- **Schemas with enumerated values** (`severity` can only be high, medium, low).
- **Nested JSON** (an object containing other objects or arrays).
- **Tables** in markdown (when the output goes to Slack or a docs page).

The technique is the same one you learned: **show the exact schema**. The difference is that now the schema is more complex.

### Pattern 1: array of objects

```
<output_format>
  [
    { "field1": "string", "field2": "number" }
  ]
</output_format>
```

Showing the array with ONE object inside is enough. The model understands it should replicate the structure for each item.

### Pattern 2: enumerated values

```
"severity": "high|medium|low"
```

The `|` symbol tells the model the value must be one of the three. Works better than saying "high, medium or low priority" in prose.

### Pattern 3: the empty-array case

If the input might have no items to extract, **say it explicitly**:

```
If there are no problems, return `[]`.
```

Without that line, the model will:

- Invent an empty object `[{}]` (worse).
- Return a prose message saying there are no problems (breaks the parser).
- Invent a plausible problem.

### Validation: deterministic + judge

When you ask for structure, you can validate a lot with regex/json-parse:

- Does it start with `[`?
- Is it valid JSON?
- Does each item have the right keys?

And you leave only two things to the judge:

- Are the values correct for the input?
- Does it cover every item from the input (coverage)?

> The more deterministic your validation, the more reliable the feedback. Save the judge for what only a model can measure.

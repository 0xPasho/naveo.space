## Big task, big problems

When you ask the model *"summarize, classify, and recommend an action"* in one sentence, it will:

1. Mix all three into a paragraph.
2. Forget one (usually the last).
3. Generate them in different order or format each time.

The fix: **break the task into explicit sub-steps inside the prompt**.

### CoT vs decomposition: similar but different

| | CoT (lesson 14) | Decomposition |
| --- | --- | --- |
| When | One task that requires deep reasoning | Multiple parallel tasks |
| Structure | Thought steps → conclusion | N labeled outputs |
| Example | "Analysis: ... / Cause: ..." | "SUMMARY: ... / SEVERITY: ... / ACTION: ..." |

CoT is **vertical** (think deep about one thing). Decomposition is **horizontal** (do N separate things).

### How to decompose in 3 steps

**1. List the sub-tasks explicitly** in the instruction:

```
This task has 3 parts:
1. Summarize the issue (one sentence).
2. Classify severity (high, medium, low).
3. Recommend a concrete action.
```

**2. Show the exact output format** with labels:

```
SUMMARY: ...
SEVERITY: high|medium|low
ACTION: ...
```

**3. Make each part parseable** from code.

If SUMMARY is always on its own line, you can extract it with a one-line regex. If it's buried in a paragraph, you need another LLM just to parse it. That multiplies cost and error.

### Why it works

Each label is a "commitment" by the model. When it writes `SEVERITY:`, the next-token probabilities collapse onto `high|medium|low`. There's no room to wander.

Without labels, the model generates a paragraph where "severity" might appear at the start, middle, end, or not at all. With labels, it's where it should be, in the format you need.

> The hidden bonus: decomposing also makes **debugging easier**. If only severity is wrong, you know exactly where to look. Without decomposition, "the prompt is broken" is all you've got.

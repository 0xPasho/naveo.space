## Few-shot: two questions, not one

You've already seen **what** few-shot is and **why** it works. Now the two remaining questions: how many examples? which ones?

### How many: 2 to 5 usually wins

Too few: the model doesn't catch the pattern. Too many: it starts fixating on surface stuff (specific words, length, punctuation) instead of the concept.

Practical rule:

| Count | When to use |
| --- | --- |
| 1 example | Only if the task is trivial (very clear format, one rule) |
| 2-3 examples | Sweet spot for simple classification, data extraction |
| 4-5 examples | Tasks with nuance or fuzzy categories |
| 6+ examples | Rarely helps. Cost grows, the model gets confused |

### Which ones: coverage > redundancy

Three examples of the same type teach less than one example per type.

**Poorly selected:**

```
- urgent: "reactor 102%"
- urgent: "leak in bay 7"
- urgent: "emergency in dock"
```

The model "sees" 3 examples but they all say the same thing. It learns that **everything is urgent**. When a normal message arrives, it'll classify as urgent because that's what it saw.

**Well selected:**

```
- urgent: "reactor 102%, venting now"
- normal: "monthly diagnostics Thursday 09:00"
- spam: "FREE crew tokens. claim before midnight"
```

Each example covers a **different category**. The model learns the difference, not the similarity.

### Bonus: include the fuzzy case

If your classifier is going to fail on any case, it'll fail on the **edges**: the message that looks urgent but isn't, the one that looks like spam but is legit. If you know what that edge is, **add it as an extra example**.

```
- normal: "Bru: hey, can we check the bay 2 panel today or wait for tomorrow's shift?"
```

That message **looks** urgent (mentions bay, asks a direct question), but its real priority is normal. An example like that teaches the model not to fall for *"any mention of bay = urgent"*.

> Rule: 3-5 examples, covering every category, plus 1 fuzzy case if you know it. More valuable than 10 examples of the easy case.

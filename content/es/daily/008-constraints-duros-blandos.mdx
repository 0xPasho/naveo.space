  - kind: prompt-AB
    question: |
      Orbit te pide ayuda con un prompt: el resumen tiene que entrar en
      una notificación push de máximo 50 palabras. ¿Cuál de las dos
      formulaciones le da el output que necesita?
    optionA: |
      ```
      Trata de ser breve y conciso. El resumen es para una notificación
      push, así que mantenelo corto.
      ```
    optionB: |
      ```
      MAX 50 palabras. Cortar antes que pasarse. Si no entra en 50,
      eliminá detalles secundarios. Devolveme solo el resumen, sin
      contar palabras al final.
      ```
    correct: B
    explanation: |
      El A es un constraint blando: "trata", "conciso", "corto". El
      modelo va a interpretar eso como una sugerencia y te puede
      devolver entre 30 y 120 palabras, dependiendo del contenido.
      Algunos van a entrar en la notificación, otros no.

      El B es duro: dice el numero exacto, dice qué hacer si no entra
      ("cortar antes que pasarse", "eliminar detalles secundarios"), y
      previene un efecto comun ("sin contar palabras al final" evita que
      el modelo agregue "Total: 47 palabras" abajo).

      Regla: si el constraint es operacional (entra o no entra), tiene
      que ser duro. Si es preferencia (mejor mas corto, pero no se
      rompe nada si es largo), puede ser blando.

  - kind: prompt-AB
    question: |
      Le pediste al modelo: "MAX 30 palabras, no te pases por nada". El
      reporte original tiene un detalle critico de seguridad que no
      entra en 30 palabras. ¿Cuando un constraint duro es
      contraproducente?
    optionA: |
      ```
      Mantené el constraint duro de 30 palabras. Si no entra todo,
      mejor que el modelo decida qué cortar.
      ```
    optionB: |
      ```
      Subi el limite a 60 palabras o agregá una excepcion: "MAX 30
      palabras EXCEPTO si hay detalles de seguridad, en cuyo caso
      mencionarlos completos."
      ```
    correct: B
    explanation: |
      Un constraint duro es util cuando la regla es razonable para todos
      los inputs. Cuando hay un input donde la regla compite con algo
      mas importante (seguridad, accuracy, compliance), el modelo tiene
      dos malas opciones: respetar el constraint y omitir info critica,
      o ignorar el constraint y romper tu confianza.

      Cuando esto pasa repetido, el constraint duro empieza a fallar de
      formas raras: el modelo "rompe" el limite cuando le parece
      importante, o trunca el detalle critico sin avisar. Ninguna es
      buena.

      Solucion: o subis el limite a uno que cubra el peor caso, o
      escribis la excepcion explicitamente. "MAX X palabras EXCEPTO en
      caso Y" es legal y el modelo lo respeta mejor que un limite
      universal contra el que pelea cada tanto.
---

Misión del día: el constraint duro es tu cinturón de seguridad para lo
crítico. El blando es para el estilo. Si los confundís, el modelo gana
y vos perdés.

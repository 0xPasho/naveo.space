  - kind: prompt-AB
    question: |
      Atlas te muestra dos descriptions para una tool `charge_card`. Hay
      un retry automático arriba en el cliente. ¿Cuál description hace
      que el modelo no la dispare dos veces sin querer?
    optionA: |
      ```
      Cobra una tarjeta. Crea un nuevo cargo cada vez que se invoca.
      ```
    optionB: |
      ```
      Cobra una tarjeta. NO es idempotente: cada invocacion crea un
      cargo nuevo. Si necesitas reintento seguro, pasa
      `idempotency_key` (string unico). Con la misma key, devuelve el
      cargo previo en vez de crear uno nuevo.
      ```
    correct: B
    explanation: |
      La A es la trampa clasica. "Crea un nuevo cargo cada vez" suena
      explicativo, pero le esta diciendo al modelo (y al cliente que
      orquesta reintentos) que esta tool no se puede reintentar nunca.
      El primer timeout de red y el usuario paga dos veces.

      La B nombra el riesgo ("NO es idempotente") y ofrece el escape:
      una key que vos generas y mandas. Con la misma key, la tool
      devuelve el cargo existente. El modelo aprende a usar la key
      cuando hay chance de retry, y los reintentos dejan de ser fatales.

      Regla: side effects irreversibles (cobrar, mandar email, borrar)
      necesitan o ser idempotentes por diseño, o aceptar una
      idempotency_key. Si no, cualquier red flaky te arruina el dia.

    task: |

    steps:

        label: "Identificá el side effect concreto que la tool produce."

Misión del día: si la tool tiene side effect, decile al modelo si reintentar
es seguro. El silencio en la description es la causa más cara de bug.

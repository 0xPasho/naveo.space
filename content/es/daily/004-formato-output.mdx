  - kind: prompt-AB
    question: |
      Vega le pide al modelo el resumen de un turno para guardar en la base.
      Otro servicio va a leer el resultado. ¿Qué prompt mandás?
    optionA: |
      ```
      Resumime el turno y devolveme la info importante de forma clara.
      ```
    optionB: |
      ```
      Devolveme JSON exacto, sin texto antes ni después:
      {
        "shift_id": string,
        "events_count": number,
        "incidents": [{ "id": string, "severity": "low"|"med"|"high" }]
      }
      ```
    correct: B
    explanation: |
      Si el output lo consume otro programa, "de forma clara" no alcanza. El
      modelo puede devolverte bullets, prosa, o JSON con prefijo tipo
      "Aca esta el JSON:". Cualquiera de esas rompe el parser del lado que
      consume.

      El B no deja espacio. Define el schema campo por campo, dice "sin
      texto antes ni despues" para matar el preambulo, y los enum
      ("low"|"med"|"high") evitan que el modelo invente categorias nuevas.

      Regla: si va a maquina, schema explicito. Si va a humano, el formato
      puede ser mas relajado.
  - kind: prompt-anatomy
    task: |
      Etiqueta las partes de este prompt bien armado. Fijate especialmente
      en la parte donde se especifica el formato.
    parts:
      - id: role
        label: "Rol del asistente"
        text: "Sos analista de turno. Resumis eventos operacionales en formato consumible por una API."
      - id: task
        label: "Task (lo que hay que hacer)"
        text: "Resumi el siguiente reporte de turno y extrae los incidentes mencionados."
      - id: format
        label: "Format spec (la forma exacta de la salida)"
        text: "Devolve solo JSON valido con la forma: { \"summary\": string, \"incidents\": [{ \"id\": string, \"severity\": \"low\"|\"med\"|\"high\" }] }. Sin texto antes ni despues."
      - id: fallback
        label: "Fallback (que hacer si falta info)"
        text: "Si no hay incidentes, devolve incidents: []. Nunca omitas el campo."
  - kind: prompt-AB
    question: |
      Necesitas un output predecible para guardar en base. Dos formas de
      pedirlo: "structured" o JSON con campos exactos. ¿Cual te da mas
      consistencia?
    optionA: |
      ```
      Devolveme la respuesta de forma structured, con los campos
      importantes bien organizados.
      ```
    optionB: |
      ```
      Devolveme JSON con exactamente estos campos:
      - shift_id (string)
      - events_count (number)
      - incidents (array de objetos con id, severity)
      Nada mas, nada menos.
      ```
    correct: B
    explanation: |
      "Structured" es una palabra que suena tecnica pero no compromete a
      nada. El modelo te puede devolver YAML, una tabla markdown, JSON con
      campos distintos cada vez, o un objeto anidado raro. Todas son
      "structured" segun alguna interpretacion.

      El B nombra cada campo, le pone tipo, y cierra con "nada mas, nada
      menos". El modelo no tiene espacio para improvisar. Manana corres el
      mismo prompt y obtenes el mismo schema.

      Regla operativa: cuando el output va a un parser, mencionar el
      formato no alcanza. Tenes que listar los campos uno por uno.

Misión del día: el formato lo elige el consumidor del output, no el que
escribe el prompt. JSON para máquinas, bullets para humanos rápidos,
prosa para humanos que leen.

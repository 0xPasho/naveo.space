  - kind: prompt-AB
    question: |
      Hex te muestra el catálogo de un MCP. Tiene 20 tools, pero seis
      son variantes casi iguales: `get_user`, `fetch_user`,
      `read_user_profile`, `lookup_user`, `user_by_id`, `find_user`.
      El modelo a veces invoca la equivocada. ¿Qué hacés primero?
    optionA: |
      ```
      Sumar al system prompt un parrafo que explique cuando usar cada
      una. "fetch_user es para X, get_user es para Y..." asi el modelo
      tiene contexto.
      ```
    optionB: |
      ```
      Quedarse con UNA tool `get_user` que acepte un parametro `mode`
      (basic / full / by-email). Borrar las otras cinco del catalogo.
      ```
    correct: B
    explanation: |
      La A trata el síntoma. Sumar un párrafo de "cuándo usar cuál"
      mete más texto al system prompt, infla el context, y obliga al
      modelo a razonar entre seis nombres parecidos cada vez. Algunas
      veces va a acertar, otras no.

      La B trata la causa. El problema no es que el modelo no entienda
      las descriptions: el problema es que seis tools cubren el mismo
      espacio. Plegarlas en una sola con un argumento que diga el modo
      elimina la decisión.

      Regla: cuando dos tools se confunden en la práctica, casi nunca
      la respuesta es "explicarlas mejor". Es plegarlas en una, o
      borrar la que sobra. Curaduria > documentacion extra.

ganan a veinte que el modelo no sabe distinguir.

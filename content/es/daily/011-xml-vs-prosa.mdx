  - kind: prompt-AB
    question: |
      Echo te muestra dos formas de pasarle la misma información a
      Claude: la entrada del usuario, un par de ejemplos, y la tarea.
      ¿Cuál separa mejor las piezas?
    optionA: |
      ```
      Te paso la entrada del usuario que es "el server se cayó", aca van
      algunos ejemplos: server down -> alta, ticket abierto -> media, y
      necesito que clasifiques la entrada en alta, media o baja.
      ```
    optionB: |
      ```
      <ejemplos>
        <ej><in>server down</in><out>alta</out></ej>
        <ej><in>ticket abierto</in><out>media</out></ej>
      </ejemplos>

      <entrada>el server se cayó</entrada>

      <tarea>
        Clasificá la entrada en alta, media o baja, basándote en los ejemplos.
      </tarea>
      ```
    correct: B
    explanation: |
      El A es un párrafo donde todo se mezcla: la entrada del usuario,
      los ejemplos, y la tarea final viven en la misma oración. El
      modelo tiene que parsear qué es qué antes de pensar.

      El B usa tags XML claros. Cada pieza queda en su caja. El modelo
      lee `<entrada>` y sabe que eso es lo que tiene que clasificar,
      no parte de los ejemplos.

      Regla: cuando hay más de una pieza de información, separala con
      tags. Claude está entrenado para respetar `<tags>` y los lee
      mejor que negritas o guiones.

  - kind: prompt-AB
    question: |
      Echo arma un prompt largo con instrucciones, documentación de
      producto y la pregunta del usuario. ¿Cuál estructura le da al
      modelo menos chances de confundir piezas?
    optionA: |
      ```
      Instrucciones: ayudá al cliente. Acá está la doc del producto:
      el plan free tiene 100 requests/mes, el pro 10k, el enterprise
      ilimitado. La pregunta del cliente: cuántos requests trae el pro?
      Respondé corto.
      ```
    optionB: |
      ```
      <instrucciones>
        Respondé al cliente en una sola línea, sin saludo.
      </instrucciones>

      <documentacion>
        Plan free: 100 requests/mes.
        Plan pro: 10.000 requests/mes.
        Plan enterprise: ilimitado.
      </documentacion>

      <pregunta>
        Cuántos requests trae el plan pro?
      </pregunta>
      ```
    correct: B
    explanation: |
      El A funciona en prompts cortos, pero en cuanto el bloque de
      documentación crece, el modelo empieza a confundir qué es
      "instrucción" y qué es "contenido a usar". A veces te responde
      sobre las instrucciones en vez de sobre la pregunta.

      El B aísla cada pieza. La documentación es contenido a referenciar,
      las instrucciones son comportamiento, la pregunta es lo que hay
      que responder. Sin riesgo de mezcla.

      Bonus operativo: si después necesitás cambiar solo la doc, tocás
      una etiqueta. En el A tenés que reescribir el párrafo entero.

      - id: task
        label: "<tarea> (instrucción operativa)"
        text: "<tarea>Devolvé solo la categoría: auth, billing, performance, o data.</tarea>"

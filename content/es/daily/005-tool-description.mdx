  - kind: tool-description
    toolName: send_alert
    context: |
      Schema de la tool `send_alert`:

      {
        "name": "send_alert",
        "parameters": {
          "type": "object",
          "properties": {
            "severity": { "type": "string", "enum": ["low", "med", "high", "critical"] },
            "message": { "type": "string" },
            "channel": { "type": "string", "enum": ["pager", "email", "chat"] }
          },
          "required": ["severity", "message", "channel"]
        }
      }

      `send_alert` despierta a alguien del turno on-call. No es un log:
      cada invocacion cuesta tiempo de una persona real. Escribi una
      description que diga claramente cuando SI invocarla, cuando NO, y
      que pasa al invocarla.

      El validador busca tres ideas, no frases exactas: que mencione
      "urgente" o sinonimo, que aclare que la "tripulacion" o el equipo
      on-call recibe la alerta, y que ate la decision a un "incidente"
      concreto y no a consultas hipoteticas.
    starter: |
      Manda una alerta.
    requiredPhrases:
      - "urgente"
      - "tripulación"
      - "incidente"
  - kind: prompt-AB
    question: |
      Forge te muestra dos descriptions para la misma tool `send_alert`.
      ¿Cuál hace que el modelo la elija mejor cuando toca?
    optionA: |
      ```
      Envía una alerta al equipo correspondiente.
      ```
    optionB: |
      ```
      Despierta a la persona on-call cuando hay un incidente urgente que
      requiere atencion inmediata de la tripulacion. NO usar para
      consultas hipoteticas, simulaciones, o eventos historicos. Usar
      severity "critical" solo si hay riesgo de perdida de datos o
      sistema caido en produccion.
      ```
    correct: B
    explanation: |
      La A es vaga en las tres dimensiones que importan. "Envia una
      alerta" no dice si despierta a alguien o si solo guarda un log.
      "Al equipo correspondiente" tampoco define cual ni cuando. El
      modelo no tiene como elegirla con confianza, asi que la usa de
      menos o de mas.

      La B nombra el efecto concreto ("despierta a la persona on-call"),
      el disparador correcto ("incidente urgente"), los casos NO
      ("consultas hipoteticas, simulaciones, historicos"), y guia el uso
      de severity. El modelo ahora puede decidir bien.

      Regla: la description tiene que decir lo que el nombre de la tool
      no comunica solo. Si tu description repite el nombre, no esta
      aportando.

Misión del día: la description es la única forma que tiene el modelo de
elegir tu tool. Si no le decís qué pasa, cuándo SÍ y cuándo NO, no la
elige.

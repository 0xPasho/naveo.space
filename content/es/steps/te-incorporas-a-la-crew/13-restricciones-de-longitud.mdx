## Por qué los modelos son malos para contar

Los LLMs **no cuentan bien**. No tienen una calculadora interna. Cuando les decís "máximo 50 palabras", están aproximando, no midiendo. Por eso las restricciones de longitud necesitan más cuidado que otras.

### Tres restricciones que funcionan mejor que "máximo N palabras"

**1. Caracteres en vez de palabras.**

```
"Máximo 120 caracteres."
```

Mejor que palabras porque los modelos son más consistentes con la unidad. Aún así no son exactos, pero el error es ±20% en vez de ±50%.

**2. Estructura forzada en vez de longitud.**

```
"Exactamente una oración."
"Una lista de 3 ítems, ni más ni menos."
"Dos párrafos: el primero el qué, el segundo el porqué."
```

La estructura es más fácil de respetar que un número. Si tu prompt dice "exactamente 3 bullets", el modelo casi siempre devuelve 3.

**3. Prohibiciones específicas.**

```
"NO uses signos de exclamación."
"NO incluyas explicaciones después de la respuesta."
"NO uses listas con bullets."
```

Cada prohibición específica corta una rama común de "elaboración" del modelo.

### El truco del andamio

Si querés exactamente una oración, **terminá el prompt con una pista visual**:

```
Notificación: 
```

(con el espacio y los dos puntos, sin valor). El modelo trata eso como el "andamio" sobre el cual escribir y tiende a respetar el formato.

### Por qué fallan las longitudes vagas

*"Sé conciso"* es completamente subjetivo. *"Una oración"* es preciso. La diferencia está en lo que el modelo puede chequear contra su output mientras genera. Reglas que se pueden verificar en tiempo real funcionan; reglas estéticas no.

> Validación práctica: usá regex para chequear estructura (cantidad de puntos, presencia de bullets, saltos de línea). Usá longitud entre N y M caracteres como límite duro. Dejá el judge para la calidad del contenido.

A la derecha, tres eventos de distinto tamaño. Escribí un prompt que devuelva una sola oración corta para los tres, incluyendo el caso del input multi-evento.

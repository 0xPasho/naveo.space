## El modelo es un perro grande: simpático pero charlatán

<Character name="forge" /> tiene una regla simple en su pipeline: el output del modelo entra **directo** a `JSON.parse(...)`. Sin limpieza, sin regex previos, sin "quitemos los backticks que mete a veces".

Eso significa que outputs como estos revientan el sistema:

- **Output con preamble:** primero el modelo escribe "Aquí está el JSON que pediste:", después el JSON en la línea de abajo. `JSON.parse` ve la primera línea y muere.
- **Output envuelto en bloque markdown:** el modelo agrega tres backticks seguidos de `json` antes del JSON y otros tres backticks después. `JSON.parse` lee el primer backtick y muere.
- **Output con cierre conversacional:** el JSON viene en medio, con texto antes ("Claro, basado en el reporte:") y después ("¿Necesitas algo más?"). `JSON.parse` muere en el primer carácter no-JSON.

Los tres tienen el JSON correcto adentro. pero el parser solo ve la primera línea y se cae. Tu trabajo: **forzar al modelo a callarse**.

### Tres técnicas que funcionan

1. **Decirlo explícito en el prompt.** "Responde SOLO con el JSON, sin texto antes ni después." Suena obvio, pero la mayoría de prompts no lo dicen y el modelo defaultea a ser servicial.

2. **Mostrar el primer carácter.** Una línea como "Tu output empieza con llave abierta y termina con llave cerrada. Nada más." ata al modelo a una restricción sintáctica, no una pragmática. Funciona mucho mejor.

3. **Prohibir el code fence específicamente.** Claude tiende a envolver JSON en bloques markdown. Si tu pipeline no los espera, escribilo: "Sin code fences. Sin triple backtick. Solo el JSON crudo."

### Cómo se evalúa

6 criterios deterministas + 1 judge:

1. El output empieza con llave abierta.
2. El output termina con llave cerrada.
3. No contiene triple backtick en ningún lado.
4. Es JSON parseable.
5. Tiene las 3 llaves correctas.
6. Los valores son los correctos para cada reporte.

Los 3 casos tienen que pasar **todos** los criterios. El judge ya es barato porque si fallaste lo determinista, no se invoca.

> Tip: pasá primero los criterios deterministas. Cuando todos esos estén verdes, el judge casi nunca falla. el modelo ya está siendo ordenado.

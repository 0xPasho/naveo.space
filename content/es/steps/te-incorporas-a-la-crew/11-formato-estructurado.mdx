## Más allá del JSON simple: estructuras anidadas y schemas

Ya escribiste un prompt que extrae un objeto plano (`alias`, `puesto`, `contacto`). La realidad es que muchas tareas necesitan estructuras más ricas:

- **Listas** de objetos (varios items del mismo tipo).
- **Schemas con valores enumerados** (`severity` solo puede ser high, medium, low).
- **JSON anidado** (un objeto que contiene otros objetos o arrays).
- **Tablas** en markdown (cuando el output va a un Slack o un docs).

La técnica es la misma que aprendiste: **mostrar el esquema exacto**. La diferencia es que ahora el esquema es más complejo.

### Patrón 1: array de objetos

```
<output_format>
  [
    { "campo1": "string", "campo2": "number" }
  ]
</output_format>
```

Mostrar el array con UN objeto adentro es suficiente. El modelo entiende que debe replicar la estructura por cada item.

### Patrón 2: valores enumerados

```
"severity": "high|medium|low"
```

El símbolo `|` indica al modelo que el valor debe ser uno de los tres. Funciona mejor que decir "alta, media o baja prioridad" en prosa.

### Patrón 3: el caso del array vacío

Si el input puede no tener items que extraer, **decilo explícitamente**:

```
Si no hay problemas, devuelve `[]`.
```

Sin esa línea, el modelo va a:

- Inventar un objeto vacío `[{}]` (peor).
- Devolver un mensaje en prosa explicando que no hay problemas (rompe el parser).
- Inventar un problema plausible.

### Validación: deterministic + judge

Cuando pedís estructura, podés validar mucho con regex/json-parse:

- ¿Empieza con `[`?
- ¿Es JSON válido?
- ¿Cada item tiene las claves correctas?

Y dejás solo dos cosas al judge:

- ¿Los valores son correctos para el input?
- ¿Cubre todos los items del input (cobertura)?

> Cuanto más determinista la validación, más confiable el feedback. Reservá el judge para lo que solo un modelo puede medir.

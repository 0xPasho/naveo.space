## El agente que sabe cuándo parar.

<Character name="orbit" /> tiene una regla: *cualquier loop sin condición de salida es un incidente esperando suceder*. Si vas a dejar al agente correr solo, tenés que poder responder antes de presionar **Run**: ¿qué lo hace parar?

Un agente que llama tools en loop es la pieza central de cualquier sistema autónomo moderno. Suena simple: modelo elige tool, ejecutás, le mandás el resultado, repetís hasta que termine.

El problema: **¿cuándo "termina"?** Tres formas de no terminar nunca:

1. **Loop infinito.** El modelo se "atasca" llamando el mismo tool una y otra vez con los mismos args.
2. **Loop largo pero no infinito.** El modelo va haciendo cosas razonables pero no converge. 30 turnos después seguís consumiendo tokens.
3. **Error que cascadea.** Un tool falla, el modelo no se entera, sigue llamando cosas que dependían de ese resultado.

Diseñar bien las **stop conditions** es lo que separa un agente que se puede dejar correr de uno que necesita supervisión humana constante.

### Las tres stop conditions

1. **`max_steps`**. contador absoluto. Si llega a 10-15 sin un `final`, salí con error. Salvavidas universal.
2. **Loop detection**. si el modelo pide el mismo tool con los mismos args 2-3 veces seguidas, está atascado. Salí.
3. **Tool errors propagados**. cuando un tool falla, el error vuelve al modelo como input (`{ ok: false, error: ... }`). El modelo decide cómo recuperarse. El loop **NO se rompe**. solo se rompe si el modelo lo decide o si las otras stop conditions disparan.

### El task

Implementá el loop. A la derecha tenés un starter que **funciona en el happy path** pero no tiene ninguna stop condition.

### Por qué no ejecutamos tu código

Mismo principio que step 06 de Track 3: un LLM-judge lee tu implementación y razona si manejaría correctamente los 4 escenarios. Eso significa que tu código tiene que ser **leíble**. Nombres claros, control flow obvio, early returns para los errors.

### Cómo se evalúa

5 criterios LLM-judge:

1. `max_steps` implementado correctamente.
2. Loop detection implementado.
3. Errores de tool vuelven al agente como input, no terminan el loop.
4. Happy path intacto.
5. Returns estructurados con `ok` boolean.

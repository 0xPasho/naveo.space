    Estás armando un asistente que ayuda a la crew a redactar partes. Hay
    dos formas de meter "siempre respondé en formato bullet list con
    timestamps UTC" en el sistema. ¿Cuál es la correcta?
  optionA: |
    System prompt:
    "Eres un asistente útil."

    User message:
    "Ayudame a redactar este parte. Siempre respondé en formato bullet
    list con timestamps UTC. Acá va el parte: [contenido]."
  optionB: |
    System prompt:
    "Eres un asistente que ayuda a redactar partes para la crew. Tus
    respuestas son siempre bullet lists con timestamps en formato UTC
    (HH:MM). Nunca devuelvas prosa corrida."

    User message:
    "Ayudame a redactar este parte: [contenido]."
  correct: B
  explanation: |
    Las dos llegan al mismo modelo, pero **van a contextos diferentes** y
    el modelo las trata distinto.

    El **system prompt** es para reglas que aplican a TODA la conversación
    sin importar qué te pida el usuario después: rol, formato global,
    restricciones permanentes. El modelo le da más peso y lo trata como
    "configuración del agente".

    El **user message** es la tarea específica del turno actual: el
    contenido a procesar, la pregunta puntual.

    El A funciona — el modelo va a obedecer — pero tiene tres problemas
    cuando creces:

    1. **Multi-turn se rompe.** En el segundo turno, el usuario solo manda
       "y ahora resumilo en una línea". Si la regla del formato vivía en
       el user message del turno 1, el modelo la fue olvidando turno a
       turno. En el system prompt, persiste.
    2. **No es separable.** No podés cambiar la regla del formato sin
       tocar cada user message. En el B, cambiás una sola string.
    3. **El usuario puede sobreescribir más fácil.** Las reglas en user
       messages se diluyen más rápido si el usuario las contradice. Las
       reglas en system prompt aguantan más antes de ceder.

    Regla simple: **lo que aplica a toda la conversación va en system; lo
    que aplica a un turno va en user.**

## Dos slots, dos propósitos

Hasta ahora cada interacción que viste fue de un solo turno: un prompt entra, una respuesta sale. En multi-turn el modelo recibe **dos cosas en cada llamada**:

- Un **system prompt** — único para toda la conversación, define quién es y cómo opera el modelo.
- Una **cadena de user/assistant messages** — los turnos de la conversación que va escalando.

Confundir cuál cosa va en cada slot es uno de los bugs más comunes de los primeros pipelines multi-turn que arma alguien.

### La regla que rara vez falla

| Va en **system prompt** | Va en **user message** |
|---|---|
| Rol ("Eres el asistente de…") | Tarea de este turno ("Ayudame con esto") |
| Formato global de respuesta | Contenido específico a procesar |
| Restricciones que aplican siempre | Aclaraciones del turno |
| Tono, idioma, persona | Datos del turno actual |
| Reglas de seguridad / guardrails | Preguntas puntuales |

### Por qué importa

En turn 1, los dos prompts del ejercicio van a producir lo mismo. La diferencia explota en turn 5, cuando el usuario solo manda "y ahora resumilo":

- Si la regla del formato vivía en el user message del turn 1, el modelo la ve cada vez **más lejos** en el contexto. La empieza a olvidar.
- En el system prompt, la regla **persiste en cada llamada** sin importar qué tan larga sea la conversación.

> Otra forma de verlo: si copiás el system prompt y lo pegás como user message, técnicamente funciona — pero estás resignando la garantía de que la regla aguanta toda la conversación. No vale la pena.

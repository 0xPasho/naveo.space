## El paso del JSON plano al JSON con capas

Hasta ahora los outputs JSON tuvieron una estructura plana: tres campos al mismo nivel. La realidad es más fea — los datos vienen con **capas**: un objeto que adentro tiene un array de objetos, y cada uno con sus propios enums.

Lo que cambia con schemas anidados:

- El modelo tiene que **decidir cuántos elementos** van en cada array.
- Los enums (`incident | routine | comm`) son fáciles de violar — el modelo puede inventar `meeting` o `task` si no sos explícito.
- Las **relaciones** entre campos (en este caso, `alert_level` deriva de los `events`) hay que hacerlas explícitas o el modelo las inventa.

### Las tres trampas

1. **Conteo del array.** Si tu prompt no te asegura "un elemento por evento", el modelo va a agrupar o desagrupar a su gusto.
2. **Enums abiertos.** "type es uno de incident, routine, comm" sin más, el modelo respeta a veces. La que casi siempre funciona: list explícita + qué cae en cada categoría.
3. **Coherencia entre campos.** Si `alert_level` se calcula desde `events`, escribí la regla en el prompt. Si no, el modelo va a "sentir" el alert level y va a quedar inconsistente.

### Cómo se evalúa

| Criterio | Cómo se mide |
|---|---|
| JSON válido + top-level keys | Determinista — parser y check de keys |
| shift / alert_level enums | Regex sobre el output |
| Conteo de events correcto | Judge sobre cada test case |
| Types + alert coherente | Judge sobre todos los casos |

> El judge es estricto con el conteo. Si tu prompt agrupa "evento + follow-up" en un solo elemento, falla. Cada bullet del log es **un** evento.

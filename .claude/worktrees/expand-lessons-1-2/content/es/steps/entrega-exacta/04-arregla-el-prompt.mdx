## El primer prompt nunca funciona. Iterar es el oficio.

Hasta acá los starters que te dimos eran un punto de arranque razonable. Este no — está **a propósito flojo**. Lo dejó otro crewmate corriendo y nadie lo limpió.

Vas a hacer lo que hace alguien que vive con esto: corres el prompt, mirás los outputs reales, identificás QUÉ se rompe, y ajustás solo eso. Volvés a correr. Repetís.

### Lo que probablemente vas a ver

Sin cambiarlo, este prompt tiene 3 problemas comunes:

1. **No separa instrucción de dato** — el ping aparece colgado al final sin tags, el modelo a veces lo confunde con parte del prompt.
2. **No declara el schema** explícitamente — pide "JSON con campos" pero no muestra la forma exacta.
3. **No define los enums** de `intent` — el modelo improvisa qué significa cada valor.

Cualquiera de los tres puede romper alguno de los 3 casos. Tu trabajo es leer el output, nombrar el síntoma, atacarlo en una sola pasada.

### Cómo se itera bien

| Síntoma | Ajuste |
|---|---|
| El output incluye prosa o el JSON está envuelto | Pedí "devuelve SOLO el JSON, sin nada antes ni después" |
| `intent` se equivoca en uno o más casos | Agregá ejemplos few-shot que cubran cada categoría |
| `summary` se pasa de 60 caracteres | Poné el cap como número + un ejemplo del largo target |
| `from`/`to` mal asignados | Marcá explícitamente en el prompt: "El que aparece antes de → es from, el que aparece después es to" |

> No reescribas el prompt entero al primer intento. Cambiá UNA cosa, volvé a correr, y mirá si ese síntoma desapareció. Si rompes 3 cosas a la vez, no vas a saber cuál de tus cambios funcionó.

### Lo que estás entrenando

No es escribir el prompt perfecto — es **leer el output y nombrar lo que se rompió**. Esa habilidad escala a cualquier sistema en producción: el prompt original siempre tiene huecos, y el oficio es cerrarlos uno por uno con evidencia.
